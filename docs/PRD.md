# 大学人生规划模拟器 PRD

版本：2.0

日期：2026-07-21

运行平台：讯飞星辰 Agent 开发平台，发布到讯飞星火/Desk

## 1. 产品概述

### 1.1 产品定位

大学人生规划模拟器是一款面向大学生的自然语言规划 Agent。它通过游戏化探索帮助用户认识自己、比较方向，再把方向转化为主规划、学期任务、行动证据、成长复盘和真实履历。

产品不是一次性生成“完美答案”的规划书，而是持续循环：

```text
认识自己 → 低成本探索 → 选择方向 → 制定规划
→ 执行动作 → 保存证据 → 复盘修正 → 形成履历
```

### 1.2 核心价值

- 把模糊焦虑变成可回答的问题和可执行的小步骤。
- 先模拟、再试错、后承诺，降低重大选择成本。
- 把用户自述、Agent 推断和真实行为证据分开。
- 让规划随着任务事实和复盘更新，而不是生成后闲置。
- 把完成过的真实事情沉淀为可核验履历素材。

### 1.3 产品原则

1. 用户只输入自然语言，不学习工作流参数。
2. 关键写入先展示草稿，再由用户明确确认。
3. 模糊的“好/继续”不等于正式确认。
4. 娱乐模拟不覆盖正式画像和主规划。
5. 失败、空数据和成功必须清楚区分。
6. 政策性内容需要知识来源和官方核验提示。
7. 不给录取、上岸、就业或收益成功概率。
8. 不把 Agent 推断写成用户事实。
9. 不伪造履历、证据或完成状态。
10. 高风险请求优先停止并建议求助，不设计试错任务。

## 2. 用户与场景

### 2.1 目标用户

- 刚进入大学、不了解可选路径的学生。
- 在升学、就业、考公或留学之间犹豫的学生。
- 已有方向但任务拆解和执行困难的学生。
- 做过项目、竞赛、社团或实习但不会整理证据的学生。
- 想通过短周期实验验证一个决定的学生。

### 2.2 核心问题

- “我适合什么方向？”
- “如果选不同路径，大学几年会怎样？”
- “我现在最该做哪件事？”
- “计划总是做不完，应该坚持还是调整？”
- “我做过的事怎么变成简历素材？”
- “这个选择能不能先试七天？”

### 2.3 核心使用场景

1. 首次使用：建立画像并体验一种探索。
2. 方向形成：基于画像、模拟/测试和知识库形成五路径建议。
3. 规划确认：比较方向，生成可修改的主规划草稿。
4. 每周执行：创建、更新、记录或完成任务。
5. 阶段复盘：根据已保存证据判断继续、调整或考虑切换。
6. 履历沉淀：把真实经历整理为待确认条目。
7. 困难决策：即时分析或启动七天低成本试错。

## 3. 产品架构

### 3.1 平台边界

- 只使用讯飞星辰 Agent 平台内能力。
- 最终用户入口发布到讯飞星火/Desk。
- 不要求用户或运行时调用方使用 API。
- 子业务工作流发布为同账号 MCP Server。
- MAIN 工作流使用 Agent 智能决策节点动态调用内部 MCP 工具。

### 3.2 逻辑架构

| 层 | 组件 | 职责 |
|---|---|---|
| 入口层 | MAIN-00 | 会话档案、自然语言理解、工具选择、连续调用、最终回复 |
| 画像探索 | WF-01～WF-04 | 画像、虚拟大学、生存测试、五路径推荐 |
| 规划执行 | WF-05～WF-07 | 方向与主规划、任务与行动、复盘与收束 |
| 事实与试错 | WF-08～WF-09 | 履历证据、决策与七天试错 |
| 数据层 | 11 张数据库表 | 业务状态、版本、证据与回读 |
| 知识层 | KB-01 | 五路径规则、材料、风险和核验渠道 |

### 3.3 调用规则

- 用户只使用 MAIN。
- MAIN 可按用户同一请求连续调用多个内部工具。
- 不设置每轮最多一张或两张的业务限制。
- 平台硬限制为最多 30 个工具和最大 100 推理轮次。
- 子工作流不得继续调用其他工具或工作流。
- 等待确认、等待回答、缺输入、失败、安全状态必须停止连续调用。

## 4. 会话和用户数据

### 4.1 user_key

MAIN 在新对话首轮生成一个用户不可见的 `user_key`，写入变量存储器。同一原对话后续轮次复用它。所有数据库表显式保存并按它筛选。

平台自动 `uid` 不作为业务身份，因为当前工作流页面没有把终端账号 ID 暴露成可供 SQL 引用的可信变量。

### 4.2 恢复行为

| 场景 | 产品行为 |
|---|---|
| 退出后重新打开仍存在的原对话 | 恢复同一会话变量并继续原档案 |
| 新建对话 | 新建独立档案 |
| 删除原对话 | 当前版本不能自动找回旧档案 |
| 平台清理会话变量 | 当前版本不能凭账号恢复 |

官方没有公开发布后会话变量的固定保存期限，因此产品不承诺永久记忆。

### 4.3 数据事实等级

| 等级 | 说明 | 示例 |
|---|---|---|
| `self_reported` | 用户自述 | “我觉得自己沟通一般” |
| `agent_inferred` | Agent 基于材料推断 | “可能更偏好结构化任务” |
| `behavior_verified` | 有行动或材料证据 | 提交记录、反馈、作品链接 |

Agent 推断不能升级为已验证事实；履历正式条目优先使用行为证据。

## 5. 功能需求

### 5.1 FR-01 用户画像

- 接收自然语言介绍并生成结构化画像草稿。
- 支持用户继续补充或修改同一草稿。
- 展示事实、不确定项和 Agent 推断。
- 用户明确说“确认保存这份画像”后正式保存并回读。
- 用户不复制确认 token。

### 5.2 FR-02 虚拟大学

- 基于已确认画像生成有取舍的大学情景事件。
- 每轮只展示并处理一个事件。
- 事件包含时间、精力、预算或机会成本。
- 无效回答不推进状态。
- 完成后形成选择偏好和能力线索，不覆盖正式画像。

### 5.3 FR-03 生存大冒险

- 基于画像生成多轮情境问题。
- 每轮只处理一道题。
- 结算形成五路径和能力维度信号。
- 完成后写入探索评估。
- 未完成时保存 pending 问题以便同一对话续接。

### 5.4 FR-04 五路径推荐

- 逐一评估保研、考研、就业、考公、留学。
- WF-02、WF-03 可任选其一或都完成。
- 只有一种探索证据时继续生成但降低置信度。
- 两种证据都无时提示先体验一个探索。
- 等级只使用“高匹配/中匹配/待验证/当前不建议投入”。
- 每条路径包含要求、差距、优先行动、证据、限制、备选和来源说明。
- 不提供成功概率。

### 5.5 FR-05 方向与主规划

- 读取画像和最新五路径建议。
- 对 2～3 个候选方向生成统一维度比较。
- 在同一工作流生成四层主规划：长期方向、学年目标、学期里程碑、近期行动。
- 支持草稿修改、取消和明确确认。
- 确认新版本时保留历史，不静默覆盖。

### 5.6 FR-06 任务与行动

- 支持查询、创建、更新、记录和完成。
- 任务类型包含学习、项目、科研、求职、习惯、运动、财务和生活。
- 行动日志与任务定义分表，允许一项任务多次记录。
- 完成任务需要实际证据；没有证据只记录进展。
- 延期需要新时间或下一检查点和原因。
- 取消只改状态，不删除历史。

### 5.7 FR-07 复盘与收束

- 读取 active 主规划、任务和行动日志。
- 区分事实、证据缺口和 Agent 推断。
- 输出继续、调整任务或考虑切换三类建议。
- 生成用户可读复盘和 Agent 续接摘要。
- 不直接覆盖主规划；需要改变时回到规划工作流生成新草稿。

### 5.8 FR-08 履历证据

- 从用户真实经历提取背景、任务、行动、结果和证据。
- 缺本人行动或结果时返回补充问题。
- 明确伪造请求进入安全拒绝。
- 先生成 pending 条目，支持修改/取消。
- 明确确认后正式保存并回读。

### 5.9 FR-09 决策与七天试错

- 即时分析可以不写数据库。
- 用户可选择把建议变成七天低成本计划草稿。
- 明确确认后计划才变为 active。
- 支持每日记录、证据、困难和下一步。
- 证据足够后生成复盘；不足时提示继续记录。
- 支持主动停止并保留历史。
- 高风险健康、安全、违法或不可逆行为不进入试错。

## 6. 对话与交互

### 6.1 用户不需要懂的内部信息

MAIN 不要求用户输入：JSON、uid、user_key、时间、assessment_id、plan_id、task_id、trial_id 或 token。内部业务键由数据库状态和代码节点确定。

### 6.2 确认语言

有效：

- “确认保存这份画像”
- “确认采用这个主规划”
- “确认保存这条履历”
- “确认启动七天试错”

无效：

- “好的”
- “继续”
- “就这样”

模糊确认需要追问，不能默认写入。

### 6.3 回复结构

每轮最终回复按顺序：

1. 本轮结果。
2. 关键依据或已保存状态。
3. 不确定项/失败说明。
4. 用户下一句可以怎么说。

不展示内部工具名、result_json、user_key、SQL 或思考过程。

## 7. 数据设计

11 张表：

1. `user_profiles`
2. `simulation_states`
3. `route_assessments`
4. `parallel_versions`
5. `main_plans`
6. `semester_tasks`
7. `growth_reviews`
8. `resume_entries`
9. `decision_trials`
10. `action_logs`
11. `session_recaps`

每张表含 `user_key:String`。创建顺序使用平台自动 `create_time`，业务状态使用版本字段。关键写入后回读。

完整字段见 [DATABASE-SCHEMA.md](database/DATABASE-SCHEMA.md)。

## 8. 非功能需求

### 8.1 性能

- 子工具输出保持五字段紧凑 JSON。
- 子工作流不嵌套。
- 不在工具之间传完整业务大对象；后续工具从数据库读取。
- MAIN 连续调用以任务需要和停止条件为准，不为用满轮次而调用。

### 8.2 可靠性

- 每个数据库节点检查 `isSuccess`。
- 空记录与失败分开。
- 模型输出先提取再校验。
- 每个分支器有默认出口。
- 写入失败和回读不一致时不声称成功。

### 8.3 安全与隐私

- 所有 SQL 按 user_key 隔离。
- user_key 不进入用户回复。
- 不把其他用户数据送给模型。
- 不保存不必要的敏感信息。
- 高风险请求停止并提供适当求助建议。
- 产品不宣称账号级认证或永久恢复。

## 9. MVP 范围

### 必须实现

- MAIN 自然语言入口。
- 同一原对话 user_key 续接。
- 九个内部 MCP 工具。
- 画像草稿/确认。
- 至少一种 WFB 探索和五路径推荐。
- 主规划草稿/确认。
- 任务、行动证据和复盘。
- 真实履历草稿/确认。
- 即时决策和七天试错状态机。
- 11 张表、KB-01、完整调试指南和自动校验。

### 暂不实现

- 外部 API 运行时。
- 跨新对话账号级档案恢复。
- 校务系统、招聘网站或日历自动同步。
- 自动抓取实时政策。
- 多人共同编辑一个档案。
- 医疗、法律、投资专业决策。

## 10. 成功指标

- 用户首次对话能在三轮内形成可确认画像。
- 画像确认后能自然进入任一探索，而不需要复制参数。
- MAIN 能正确执行至少一个双工具连续调用用例。
- 所有等待状态都能阻止越权后续写入。
- 两个新对话不能互相读取业务数据。
- 同一原对话退出再打开后能继续。
- 任务完成记录包含真实证据。
- 履历正式条目无虚构字段。
- 写入/回读失败的回复不包含“已保存”。

## 11. 上线验收

1. 仓库四个自动校验命令全部通过。
2. WF-01～WF-09 分别在画布完成正常与失败路径调试。
3. 九张工作流发布为同账号 MCP Server。
4. MAIN Agent 智能决策只添加九个内部工具。
5. 新对话首次建档通过。
6. 同一轮“确认画像 + 开始虚拟大学”双工具通过。
7. “记录任务完成 + 复盘”双工具通过。
8. 模糊确认停止通过。
9. 工具失败停止通过。
10. 原对话恢复通过。
11. 新对话隔离通过。
12. 发布审核后在真实星火/Desk 入口复测，并在 Trace 留证。

## 12. 风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 平台未提供可信终端账号变量 | 无法账号级跨会话恢复 | 使用原对话 user_key；明确产品边界；等待平台能力 |
| 变量存储期限未公开 | 长期恢复不确定 | 不承诺永久；关键业务事实写数据库 |
| MCP 非流式和连续模型调用超时 | 一轮可能失败 | 子工具紧凑输出、禁止嵌套、停止无价值调用 |
| 模型误选工具或改写参数 | 路由或数据错误 | 精确工具描述、可信 payload、子工具格式/前置校验、Trace 测试 |
| 模型结构输出不稳定 | 写入错误 | 单输入提取、字段校验、默认失败出口、最多一次格式重试 |
| 发布审核导致验证延迟 | 不能当天验证真实入口 | 先完成画布/MCP 调试；审核后执行固定验收套件 |

## 13. 产品边界声明

本产品提供规划探索、任务组织和反思辅助，不替代学校、用人单位、考试主管部门、医疗机构、法律或金融专业人士的正式意见。涉及政策、录取、考试、就业、健康、安全或财务的内容必须结合官方渠道和专业建议核验。
