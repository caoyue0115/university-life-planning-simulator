# 讯飞星辰工作流平台 UI 配置契约

本文记录本项目允许使用的讯飞星辰页面控件、真实输入输出和配置边界。官方文档与用户实际截图共同确认的内容写为“平台事实”；本项目为了稳定运行增加的约束写为“项目规则”。

## 1. 开始节点

### 平台事实

- 新建工作流时画布自带“开始”和“结束”。
- 开始节点有唯一默认输入 `AGENT_USER_INPUT`，类型 String，用来接收当前轮用户原始输入。
- 编辑器允许点击“+ 添加”增加其他输入，但发布到星火/Desk 后，最终用户只是在聊天框中输入自然语言。

### 项目规则

- MAIN-00 和 WF-01～WF-09 都只保留 `AGENT_USER_INPUT:String`。
- 不新增 `uid`、`user_key`、时间、session、token 或业务 ID。
- MAIN 的 `AGENT_USER_INPUT` 是用户自然语言。
- 子工作流的 `AGENT_USER_INPUT` 是 MAIN 生成的两字段扁平 JSON String。

右侧输入区最终只有一行：

| 参数名 | 类型 | 必要 | 描述 |
|---|---|---:|---|
| `AGENT_USER_INPUT` | String | 是 | MAIN：用户本轮原话；子工作流：MAIN 生成的内部包装字符串 |

## 2. 节点引用下拉框

### 平台事实

右侧输入值选择“引用”后，只能看到当前节点之前、沿有效连线可达的节点变量。数据库之后才生成的变量不能反向供数据库使用。

用户实际页面已经确认：数据库 N01 的变量列表只出现上游 `N00A 代码` 和“开始”，不会出现未来节点或未连接节点。

### 项目规则

- 所有数据库 `user_key` 必须引用 N00A 解析输出。
- 引用列表里没有预期变量时，先检查连线和上游输出声明，不在输入框手写一个同名值冒充引用。
- 所有教程都按“先拖节点 → 连线 → 配置输出 → 再选择下游引用”的顺序操作。

## 3. 大模型节点

### 页面配置

1. 左侧节点列表点击“大模型”添加。
2. 模型选择 `Spark4.0 Ultra`；若当前账号名称不同，选择已开通且支持结构化指令的最高稳定版本，并在调试记录中记下实际名称。
3. “输入”逐行添加参数名，类型选“引用”，值选择上游具体变量。
4. “系统提示词”填写角色、事实边界、状态规则和唯一输出格式。
5. “用户提示词”只放当前显式输入，用 `{{参数名}}` 引用。
6. 输出格式选 `text`，输出变量使用页面默认或教程指定的 `output:String`。
7. 状态机业务关闭“对话历史”；MAIN 的 Agent 智能决策负责整轮上下文，子工具只读显式状态。

### 项目规则

- 不让模型生成 `user_key` 以外的数据库业务键和版本号。
- 不让模型判断数据库写入成功。
- 输出必须经过变量提取器和代码校验后才能写数据库。
- 提示词必须写“只输出规定结构”，但不能只靠这句话；校验失败要有默认出口。

## 4. 变量提取器

### 平台事实

- 输入参数固定名为 `input`。
- 输入类型固定为 String。
- 输入区只有一项；输出区可以声明多个变量和类型。
- 它由模型提取语义，不是严格 JSON parser。

### 正确配置

```text
模型：Spark4.0 Ultra
输入：input｜引用｜唯一上游/output
输出：按教程逐行添加
异常处理：初次搭建关闭；需要重试时只开启一次重试
```

### 错误配置

```text
user_input｜引用｜开始/AGENT_USER_INPUT
pending_json｜引用｜数据库整理/pending_json
```

页面无法把上面两行同时配置给一个变量提取器。需要两个来源时，先添加文本处理节点，把两者拼成一个 `output:String`，再让提取器引用它。

## 5. 文本处理节点

### 平台事实

- 可选择字符串拼接或字符串分割。
- 字符串拼接可以接收多个上游文本输入。
- 规则中用 `{{变量名}}` 引用输入。
- 固定输出 `output:String`。

### 项目标准拼接方式

页面“处理方式”选“字符串拼接”，输入逐行引用需要的上游变量，规则使用带标签文本：

```text
【用户原话】
{{user_input}}

【待处理对象】
{{pending_json}}

【历史状态】
{{state_json}}
```

标签不得省略，否则模型容易把用户输入和历史 JSON 混为同一事实。

## 6. 代码节点

### 平台事实

- 当前项目页面只使用 Python。
- 输入参数名必须与 `main(...)` 形参一致。
- 输出区必须声明 return dict 的同名键及类型。
- 形参、输入或输出不一致是代码节点 500 的常见原因。

### 项目规则

- 不写 `import`。
- 所有路线返回同一组键。
- String 缺省用 `""`，Integer 用 `0`，Boolean 用 `False`，数组用 `[]`，不返回 `None`。
- 不把平台报错堆栈放进用户回复。
- 每个教程在代码块前给出输入引用表，代码块后给出输出声明表。

最小示例：

```python
def main(input):
    value = str(input) if input is not None else ""
    return {"output": value}
```

输出区：

| 参数名 | 类型 | 描述 |
|---|---|---|
| `output` | String | 规范化后的字符串 |

## 7. 分支器

### 平台事实

分支器用于条件判断。每个条件可以选择引用变量、运算符、比较类型和比较值；条件可组合。

### 项目规则

- Boolean、Integer、枚举 String 判断只用分支器。
- 与字面量比较时，比较类型选“固定值/常量”，不选“引用”。
- 每个分支器都配置默认/失败出口。
- `isSuccess` 第一条路线固定为“等于 → 固定值 → true”；默认路线是失败。
- 不能只连成功路线后让其他值悬空。

## 8. 决策节点

### 平台事实

决策节点包含模型、固定 `Query`、意图名称、意图描述、默认意图和高级配置，输出识别出的意图类别。

### 项目规则

- 只用于自然语言意图分类。
- 不用于 `isSuccess == true`、`completed == true` 或版本比较。
- 所有意图写清正例、反例和对象；保留一个默认意图进入 `needs_input`。

## 9. 数据库：自定义 SQL

### 页面配置

1. 添加“数据库”节点。
2. “模式”选择“自定义SQL”。
3. “选择数据库”选择 `university`。
4. “输入”点击“+ 添加”。
5. 参数名填写 SQL 中双花括号内的名称，例如 `user_key`。
6. 类型选“引用”。
7. 值选择 `N00A 代码：解析包装/user_key`。
8. SQL 区粘贴教程给出的完整 SQL。

固定输出：

```text
isSuccess Boolean
message String
outputList Array<Object>
```

`isSuccess=true && outputList=[]` 表示执行成功但没有记录。

### 项目规则

- 读取 SQL 必须按 `user_key` 过滤。
- 读取最新版本同时按业务版本 DESC、`create_time DESC` 排序。
- 不把平台自动 `uid` 放进输入或 WHERE。
- SQL 节点后先判断 `isSuccess`，再整理 `outputList`。

## 10. 数据库：表单处理数据

### 新增数据

1. 模式选择“表单处理数据”。
2. 数据库/数据表选择 `university / 教程指定表名`。
3. 处理模式选择“新增数据”。
4. “设置新增数据”逐行选择表字段、类型“引用”和上游变量。
5. 必须显式新增 `user_key`，引用 N00A 或后续原样透传的 `user_key`。
6. 不重复添加系统字段 `id/uid/create_time`。

### 更新数据

1. 处理模式选择“更新数据”。
2. “设置数据范围”第一行固定 `user_key｜等于｜引用｜N00A/user_key`。
3. 继续添加业务键、状态或版本条件，逻辑使用 AND。
4. “设置更新数据”逐行引用已经校验的值。

固定输出仍为 `isSuccess/message/outputList`。更新后必须再用自定义 SQL 回读，不凭 `isSuccess` 宣称内容一致。

## 11. 知识库节点

### 页面配置

- 输入固定 `query:String`，引用上游文本处理/output。
- 添加 `KB-01 大学五路径规则库`。
- 调用逻辑选择“强制调用”。
- Top K=`3`。
- Score 阈值=`0.20`。
- 当前官方文档固定输出名可能显示 `result` 或页面版本显示 `results`；本项目以用户实际页面字段为准，并在 WF-04 节点输入输出表中保持前后一致，不能同时使用两个名称。

知识库没有命中时进入明确缺资料路线；模型不能凭训练记忆补政策细节。

## 12. 变量存储器

### 平台事实

- “设置变量值”把上游变量赋给会话变量。
- “获取变量值”读取已经保存的会话变量。
- 同一对话后续轮次可继续引用。
- 新建会话或删除会话会清空变量。

### MAIN-00 项目配置

- 会话变量名固定 `user_key:String`。
- 第一轮“获取变量值”为空时才生成。
- 生成并校验后用“设置变量值”保存。
- 后续再次“获取变量值”作为 Agent 智能决策输入。
- 不把画像、主规划等大对象同时存在变量存储器；这些状态以数据库为准。

官方没有公开发布后变量的固定保留期限，文档不得写“永久”或具体天数。

## 13. Agent 智能决策节点

### 平台事实

- 支持 ReACT 策略。
- 输入参数可引用其他节点或用户输入。
- 可从已发布列表同时添加多个工具或 MCP Server 类工具，最多 30 个。
- 也可填 MCP 服务器地址，上限 3 个；本项目不用外部地址。
- 提示词区域包含角色设定、思考步骤和查询。
- 最大推理轮次最多 100。
- 输出包含思考过程和最终内容；最终字段名称以当前页面固定输出为准。

### MAIN-00 项目配置

- 工具只选当前账号发布的 WF-01～WF-09 MCP Server。
- 不添加网页搜索、代码执行、外部 MCP 地址或同功能重复插件。
- 最大推理轮次填写 `100`，不增加人为工具次数上限。
- 提示词要求携带上游不可见 `user_key`，但绝不向用户展示它。
- 每个工具的 `AGENT_USER_INPUT` 都传两字段扁平 JSON String。
- 工具返回等待、失败或安全状态时立即停止。
- 不把思考过程展示给用户，最终只输出自然语言结论和下一步。

## 14. 工作流发布为 MCP Server

官方入口：

1. 完成一张 WF 子工作流并点击右上角“调试”。
2. 所有验收用例通过后点击右上角“发布”。
3. 在发布路径中选择最下面的“MCP Server”。
4. 填写教程指定的名称、功能描述、使用场景和参数说明。
5. 提交发布。
6. 回到 MAIN-00 的 Agent 智能决策节点。
7. 点击“添加插件/添加工具”。
8. 切换到 MCP 或已发布 MCP 列表。
9. 选择当前账号刚发布的工作流。

官方 FAQ 说明：工作流发布的 MCP 只在自己的账号列表中查看；MCP 只能在工作流的 Agent 智能决策节点调用。发布列表暂时找不到时，先确认子工作流已完成发布，不用把它重建成普通插件。

## 15. 消息节点

消息节点用于中间进度提示，可引用上游变量并设置回答内容。由于 MCP 子工作流为非流式调用，本项目业务子工作流不依赖消息节点向最终用户返回核心结果；核心结果必须由结束节点返回 `result_json`。

若教程保留消息节点：

- 流式输出关闭；
- 消息之后继续连接结果整理节点；
- 消息文本与实际数据库状态一致；
- 写入未回读时不得写“已保存”。

## 16. 结束节点

### 子工作流

- 回答模式选择“返回参数，由工作流生成”或当前页面等价选项。
- 参数名 `result_json`。
- 类型 String。
- 值选择“引用”，引用公共结果整理代码节点的 `result_json`。
- 不使用固定 `workflow_finished`。
- 不引用消息节点，因为消息不提供稳定业务输出。

### MAIN-00

- 回答模式选择“返回设定格式配置的回答”。
- 回答内容引用最终整理节点的 `final_reply`。
- 思考内容留空。
- 不展示 Agent 智能决策的思考过程。

## 17. 异常处理与重试

- 首次搭建先关闭隐式异常处理，用显式 `isSuccess`、校验和分支器确认主流程。
- 大模型/变量提取器输出格式偶发失败时最多重试 1 次。
- 数据库写入不得自动重试导致重复行；写入失败先回读或返回失败。
- Agent 智能决策工具失败时停止连续调用，不换另一个业务工具掩盖错误。
- 节点支持异常出口时，异常出口连接统一失败结果节点。

## 18. 调试与 Trace

工作流画布右上角先调试单节点和完整链路。发布后真实运行记录在：

```text
发布管理 → 对应应用 → 详情 → Trace
```

端到端排错顺序：

1. MAIN 是否读取/生成同一 `user_key`；
2. Agent 智能决策实际选择了哪个工具；
3. 工具参数是否只有 `AGENT_USER_INPUT`；
4. 子工作流 N00A 是否解析成功；
5. 数据库 SQL 是否按 `user_key` 查询；
6. 分支器实际走哪条路线；
7. 子工具 `result_json` 是什么；
8. MAIN 是否遵守停止条件；
9. 最终回复是否与写入和回读一致。
